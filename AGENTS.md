# AGENTS.md

이 문서는 Claude Code(또는 다른 코딩 에이전트)가 이 리포에서 작업할 때 따르는 규약입니다.
사람도 읽고 참고하세요.

## 프로젝트가 무엇인가

Madang은 이력서(Jari 엔진) + 포트폴리오 + 지식기반(Obsidian 연동)을 하나의 콘텐츠 소스에서
관리하는 개인/조직 웹 사이트 템플릿입니다. `content/resume/*.md`가 이력서의 단일 진실이며,
여기서 HTML/PDF/DOCX가 파생됩니다.

## 콘텐츠 작업 시 원칙

1. **이력서 데이터는 `content/resume/` 아래 구조화 마크다운으로만 수정한다.**
   HTML, PDF, DOCX를 직접 손으로 고치지 않는다 — 항상 빌드로 생성한다.
2. **frontmatter는 스키마를 따른다.** `packages/jari/src/schema.ts`의 zod 정의가 진실이다.
   새 필드가 필요하면 스키마를 먼저 바꾸고, 그 다음 콘텐츠를 채운다.
3. **포지션 태그(`positions: [...]`)를 판단해서 채운다.** 새 경력/프로젝트 항목을 추가할 때,
   `content/resume/positions/*.md`에 정의된 포지션 중 어디에 관련 있는지 사용자에게 묻거나
   합리적으로 추론해서 태그를 단다. 애매하면 사용자에게 확인한다.
4. **디자인은 `design/tokens.yaml`을 통해서만 바꾼다.** CSS나 reference.docx를 직접 편집하지
   않는다 — 토큰에서 생성한다.
5. **콘텐츠 추가/수정 후에는 반드시 정적 검사를 실행해 검증한다:**
   ```bash
   npm run check   # = tsc 빌드 + vitest + madang check(스키마·포지션·위키링크 무결성)
   ```
   콘텐츠만 바꿨다면 `npm run madang -- check`(또는 `npx madang check`)만으로 충분하다.
   스키마 위반이나 링크 무결성 오류(`[[위키링크]]`가 존재하지 않는 노트를 가리킴)가 있으면
   먼저 해결한다.
6. **커밋 전에 diff를 사람에게 요약해서 보여준다.** 이력서/포트폴리오 콘텐츠는 개인정보이므로,
   자동으로 커밋하거나 푸시하지 않고 확인을 받는다.

## 일반적인 작업 흐름 예시

**"이번 프로젝트를 이력서에 추가해줘" 같은 요청을 받으면:**
1. `content/resume/projects/`에 새 md 파일 생성 (frontmatter + 요약 + 성과 불릿)
2. 관련 포지션 태그 판단 (`positions.md` 확인 후 필요하면 질문)
3. 관련 있는 노트/포트폴리오 항목이 있으면 `links:` 필드에 위키링크로 연결
4. `npm run build`로 검증
5. 변경 사항 요약 후 커밋 여부 확인

## 하지 말아야 할 것

- `dist/`의 산출물을 직접 수정하지 않는다 (빌드로만 생성)
- `madang init --force`를 사용자 확인 없이 실행하지 않는다 — `content/resume/`가 템플릿으로
  초기화되어 실제 이력서 데이터가 삭제된다
- 스키마에 없는 임의 필드를 frontmatter에 추가하지 않는다 (먼저 스키마 확장)
- 사용자 확인 없이 git push, 배포 트리거를 실행하지 않는다
- `content/notes/`(Obsidian vault)의 파일명을 임의로 바꾸지 않는다 — 위키링크와 퍼머링크가
  깨진다. 이름 변경이 필요하면 리다이렉트 처리를 함께 한다.
